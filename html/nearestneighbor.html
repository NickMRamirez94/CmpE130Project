Nearest Neighbor Algorithm

This is a simple implementation of the nearest neighbor algorithm.

Remember...this does not always (and rarely does) give the most optimal solution; however, it is very fast!

Learn more

How Does it Work?

This solution uses a very strong greedy strategy. This is because it looks at the current city and will go to the closest city that it's connected to. It will continue to do this until it visits all of the cities. This is a great strategy because the algorithm is a lot faster than a brute force approach, but it can often miss a much shorter path due to its greedy nature. Therefore, it provides a nice, yet often not optimal solution.

Running Time

The running time of the Nearest Neighbor Algorithm is O(n^2), which is fairly fast given the issue at hand. Therefore, this makes for a decent choice seeing that it will work for a large number of cities and it will give you the answer relatively quickly. Additionally, the graph shows the rate of growth for this algorithm. For small input values the algorithm still computes the solution quickly; however, fails to find the path quickly for large input values.

Quick Summary

The following steps describe the algorithm in a straightforward manner:

  1. Start at the first city and mark it visited
  2. Scan for the shortest path from the current city to another city that hasn't been visited yet
  3. Move to that city and mark it as visited
  4. Go to Step 6 if all cities have been marked as visited
  6. Loop to Step 2
  7. Return to starting city

Pros to this method:

  • Very fast
  • Straightforward implementation
  • Works for high input
  • Generally falls somewhere in between worst case and best case scenario

Cons to this method:

  • Very greedy and therefore doesn't always give optimal answer
  • Can miss a better path
